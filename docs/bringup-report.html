Hardware Chain

Arduino UnoReads PS2 receiver and prints J,select,ry,rx,ly,lx,guard.

Teensy 4.1Receives D,left,right,enable.
SabertoothsPacket serial from Teensy Serial1/TX1.

LiDAR is a separate USB device. Its only contract with the obstacle guard is to publish sensor_msgs/msg/LaserScan on /scan.

Firmware Roles

Uno Sends one line per controller packet: 
J,select,ry,rx,ly,lx,guard
The seventh guard field is latched by the Uno. START toggles it between 0 and 1.
Teensy Receives drive lines: 
D,left,right,enable
Sabertooth Chain Right address 128, left address 129, Teensy Serial1/TX1, 9600 baud.

Do not reflash automatically just because the branch changed. Reflash the Uno when you need the newer J,select,ry,rx,ly,lx,guard packet for START-button obstacle guard latching, or if it is not sending J,.... Reflash the Teensy only if it is not accepting D,....

YDLIDAR Driver Patch

The YDLIDAR driver lives outside the robot repo in /home/rudra/ros2_ws. The SDK was installed locally under /home/rudra/ros2_ws/install/YDLidar-SDK instead of system-wide.

cd /home/rudra/ros2_ws
source /opt/ros/lyrical/setup.bash
export CMAKE_PREFIX_PATH=/home/rudra/ros2_ws/install/YDLidar-SDK:$CMAKE_PREFIX_PATH
colcon build --symlink-install --packages-select ydlidar_ros2_driver

Compatibility changes made for ROS2 Lyrical and the local CMake version:

  • YDLidar-SDK CMake policies were modernized enough to configure under current CMake.
  • The SDK install step was changed to install targets directly, avoiding old build-tree LOCATION behavior.
  • ydlidar_ros2_driver links imported ROS targets instead of relying on unavailable ament_target_dependencies().
  • The driver node declares parameters with defaults, which Lyrical requires before reading YAML values.

Slamtec Driver Patch

On ROS2 Lyrical, upstream sllidar_ros2 called the old ament_target_dependencies() CMake macro. On this machine, the installed ament packages did not provide that macro file. The compatibility patch in ~/ros2_ws/src/sllidar_ros2/CMakeLists.txt used imported targets:

target_link_libraries(sllidar_node
  rclcpp::rclcpp
  std_srvs::std_srvs
  sensor_msgs::sensor_msgs
)

target_link_libraries(sllidar_client
  rclcpp::rclcpp
  std_srvs::std_srvs
  sensor_msgs::sensor_msgs
)
cd /home/rudra/ros2_ws
source /opt/ros/lyrical/setup.bash
colcon build --symlink-install --allow-overriding sllidar_ros2

Obstacle Guard

Both drive bridges create a LidarObstacleGuard. It subscribes to /scan, listens for runtime enable/disable commands on /rudra/obstacle_guard_enable, and publishes guard status on /rudra/obstacle_guard.

Keep the LiDAR driver and the obstacle guard conceptually separate. The LiDAR can keep publishing /scan for SLAM/Nav while the low-level drive filter is disabled for manual testing.

obstacle_avoidance_enabled: true
obstacle_enable_topic: "/rudra/obstacle_guard_enable"
scan_topic: "/scan"
obstacle_front_angle_deg: 70.0
obstacle_stop_distance_m: 0.45
obstacle_slow_distance_m: 1.00
obstacle_scan_timeout_sec: 0.75
obstacle_status_period_sec: 0.25

Runtime topic control:

ros2 topic pub --once /rudra/obstacle_guard_enable std_msgs/msg/Bool "{data: false}"
ros2 topic pub --once /rudra/obstacle_guard_enable std_msgs/msg/Bool "{data: true}"

PS2 START-button control:

Press START to toggle obstacle avoidance.
Requires updated Uno firmware publishing: J,select,ry,rx,ly,lx,guard

The guard field is latched by the Uno, so it stays 0 or 1 until START is pressed again. The ROS2 bridge follows that latch and reports the guard state on /rudra/obstacle_guard as enabled=0 or enabled=1. MODE is not used because the vendored PS2X library does not expose it as a normal readable button.

PS2 tuning:

ps2_obstacle_latch_enabled: true
ps2_obstacle_control_enabled: false
ps2_obstacle_control_hold_sec: 0.80
ps2_obstacle_control_threshold: 90
ps2_obstacle_control_require_stopped: true
StatusMeaning
no_fresh_scanNo recent scan. Forward commands pass through.
clearFront cone is clear. Forward commands pass through.
slowingObstacle is between slow and stop distance. Forward command is scaled down.
blockedObstacle is inside stop distance. Forward command is blocked.
reverse_or_turnReverse and turn-in-place remain available.

To temporarily get old PS2 behavior, set obstacle_avoidance_enabled: false in both sections of src/rudra_base_bridge/config/rudra_v4_hardware.yaml.

Disabling the obstacle guard does not necessarily stop the physical LiDAR from spinning. Some direct USB YDLIDAR units spin whenever powered. The runtime guard control only changes whether scan data filters motor commands.

Safety Rules

  • Lift wheels for the first motor test.
  • Keep battery power easy to remove.
  • Close Arduino Serial Monitor before launching ROS.
  • Launch only one node that owns the Teensy serial port.
  • Confirm /rudra/drive_cmd before placing the rover on the floor.
  • Start with low speed commands.
  • Do not enable Nav2/autonomy until /scan, TF, odom, and base control are boringly reliable.
