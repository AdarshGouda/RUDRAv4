Robot Overview

RUDRAv4 keeps the original split-controller design from RUDRAv3. The Uno only reads the PS2 controller and forwards compact serial packets to the NUC. The NUC runs ROS2 and decides what drive command to send. The Teensy is the only board that talks packet serial to the Sabertooths.

PS2 controllerWireless operator input.
Arduino UnoReads receiver and publishes J,select,ry,rx,ly,lx,guard.
NUC running ROS2Runs bridge nodes, publishes topics, filters forward motion using LiDAR.
Teensy 4.1Receives D,left,right,enable and drives Sabertooths.
Sabertooth driversRight address 128, left address 129, packet serial at 9600.
Drive motorsFinal rover motion output.

The LiDAR is not in the motor command chain directly. It is a separate USB sensor. Its job is to publish /scan. The ROS2 bridge consumes that scan and applies the low-level obstacle guard before sending forward motor commands.

Hardware Chain

PS2 to Uno The Uno uses the vendored PS2X library and the known working RUDRAv3 pinout: 
CLK=9
CMD=7
ATT=6
DAT=8
Uno to NUC USB serial at 115200. The Uno sends: 
J,select,ry,rx,ly,lx,guard
NUC to Teensy USB serial at 115200. The bridge sends: 
D,left,right,enable
Teensy to Sabertooths Teensy Serial1/TX1 to packet serial bus at 9600. Right controller address 128. Left controller address 129.

Manual mixing is preserved from the previous robot behavior: throttle comes from Ly, steering comes from Rx, and the bridge computes left = throttle - steering and right = throttle + steering.

Firmware and Programming

The two microcontrollers have cleanly separated jobs. Reflash only the board you need to change.

Uno firmware File: 
src/rudra_base_bridge/firmware/uno_ps2_plain_serial/uno_ps2_plain_serial.ino
Responsibility:
  • Read the PS2 receiver.
  • Publish the J,... packet over USB serial.
  • Latch the seventh guard field when START is pressed.
Teensy firmware File: 
src/rudra_base_bridge/firmware/teensy_sabertooth_serial_controller/teensy_sabertooth_serial_controller.ino
Responsibility:
  • Receive D,left,right,enable lines from the NUC.
  • Send packet serial commands to both Sabertooths.
  • Reply with ACK,... over USB serial.

Programming workflow:

  1. Open the correct sketch in Arduino IDE or Arduino IDE with Teensy support installed.
  2. Select the correct board and serial port.
  3. Upload the sketch.
  4. Close Serial Monitor before starting ROS, or it may hold the port open.
Reflash the Uno when the PS2 packet format is wrong or the START-button guard latch is missing. Reflash the Teensy when it is not accepting D,... or returning ACK,....

YDLIDAR Setup

The working LiDAR for this robot is YDLIDAR G2B. The working ROS2 stack is YDLidar-SDK plus ydlidar_ros2_driver in the external workspace at /home/rudra/ros2_ws.

On this machine, the stable known-good configuration is the LiDAR adapter's persistent /dev/serial/by-id/... path at 128000 baud, sample_rate: 5, frequency: 10.0, intensity: true, and frame laser.

Install the SDK and Driver

If you are setting up the LiDAR stack from scratch, use the external driver workspace layout below.

mkdir -p /home/rudra/ros2_ws/src
cd /home/rudra/ros2_ws/src
git clone https://github.com/YDLIDAR/YDLidar-SDK.git
git clone -b humble https://github.com/YDLIDAR/ydlidar_ros2_driver.git

Build and install the SDK into the local external workspace instead of installing it globally. That keeps the dependency isolated and reproducible for this robot.

cd /home/rudra/ros2_ws/src/YDLidar-SDK
mkdir -p build
cd build
cmake .. -DCMAKE_INSTALL_PREFIX=/home/rudra/ros2_ws/install/YDLidar-SDK
cmake --build .
cmake --install .

Then build the ROS2 driver with the SDK location on CMAKE_PREFIX_PATH.

cd /home/rudra/ros2_ws
source /opt/ros/lyrical/setup.bash
export CMAKE_PREFIX_PATH=/home/rudra/ros2_ws/install/YDLidar-SDK:$CMAKE_PREFIX_PATH
colcon build --symlink-install --packages-select ydlidar_ros2_driver
source install/setup.bash

Known Working Behavior

  • The first valid /scan can take roughly 10 to 15 seconds after launch.
  • Startup checksum warnings may appear before scan publishing begins.
  • The driver may print repeated Real points > fixed points 530 warnings while still publishing valid scans.
  • If the node is healthy, the important signal is that /scan starts publishing with frame laser.

IMU and Wheel Odom

RUDRAv4 keeps the existing PS2 and Sabertooth control chain, then layers localization on top. The Teensy still accepts D,left,right,enable commands, but the localization firmware also emits plain serial telemetry for IMU and odometry:

IMU,ax,ay,az,gx,gy,gz
ODOM,x,y,theta,linear_x,angular_z,left_vel,right_vel

The ROS 2 bridge nodes parse those lines and publish:

  • /rudra/imu/raw as sensor_msgs/msg/Imu
  • /rudra/wheel_odom as nav_msgs/msg/Odometry
  • /odometry/filtered from robot_localization

The new Teensy firmware sketch is:

src/rudra_base_bridge/firmware/teensy_sabertooth_imu_odom_controller/teensy_sabertooth_imu_odom_controller.ino

This sketch intentionally does not add PID yet. It preserves the current open-loop drive path so IMU and encoder bringup can be validated first. Add PID later only if open-loop wheel tracking proves too inconsistent.

The EKF configuration lives in:

src/rudra_base_bridge/config/localization_ekf.yaml

It currently fuses wheel odom with IMU angular velocity. That is enough to bring up odom, TF, and RViz while keeping the hardware and software stack understandable.

RViz on Aghora

Run RViz on the laptop, not on the NUC, for normal operation. The NUC only needs to publish /scan and the marker topics. Aghora subscribes over the ROS2 network.

source /opt/ros/lyrical/setup.bash
export ROS_DOMAIN_ID=42
export ROS_LOCALHOST_ONLY=0
cd /home/rudra/Projects/RUDRAv4
source install/setup.bash
ros2 launch rudra_base_bridge lidar_view.launch.py

This launch starts RViz only plus a local static TF from base_link to laser. It does not start the LiDAR driver and it does not start a drive bridge. It is a viewer, not a robot bringup launch.

The wrapper scripts in scripts/ default to ROS_DOMAIN_ID=42 and ROS_LOCALHOST_ONLY=0 if those variables are unset. That matches the current Rudra and Aghora setup and avoids the silent no-data case caused by mismatched DDS domains.

If you prefer manual RViz setup, use fixed frame base_link, add a LaserScan display for /scan, and add a Marker display for /rudra/obstacle_guard_marker.

For odometry and localization work, use bash scripts/view_localization.sh or ros2 launch rudra_base_bridge localization_view.launch.py. In that view, the fixed frame should be odom. The raw encoder odom appears on /rudra/wheel_odom and the fused estimate appears on /odometry/filtered.

The scan can appear with only the LiDAR driver running. The obstacle marker appears only when a bridge node such as ps2_uno_to_teensy or cmd_vel_to_teensy is running, because that is what creates the obstacle-guard publisher.

If RViz shows no scan even though the topic exists, set the LaserScan display reliability policy to Best Effort.

You can view odom in RViz immediately. A map is a later layer. Once odom -> base_link, base_link -> laser, and LiDAR scan data are stable, add a SLAM package to generate map -> odom.

Obstacle Guard

Both bridge nodes create the same LidarObstacleGuard. It slows forward motion inside the slow distance and blocks forward motion inside the stop distance. Reverse and turn-in-place remain available so the robot can back out.

obstacle_avoidance_enabled: true
obstacle_enable_topic: "/rudra/obstacle_guard_enable"
scan_topic: "/scan"
obstacle_front_angle_deg: 70.0
obstacle_stop_distance_m: 0.45
obstacle_slow_distance_m: 1.00
obstacle_scan_timeout_sec: 0.75
obstacle_status_period_sec: 0.25
obstacle_marker_topic: "/rudra/obstacle_guard_marker"
obstacle_marker_frame: "laser"

Runtime enable or disable:

ros2 topic pub --once /rudra/obstacle_guard_enable std_msgs/msg/Bool "{data: false}"
ros2 topic pub --once /rudra/obstacle_guard_enable std_msgs/msg/Bool "{data: true}"

PS2 guard control:

  • START toggles the latched guard bit in the Uno firmware.
  • The seventh packet field is guard, which stays at 0 or 1 until START is pressed again.
  • The ROS2 bridge follows that latch and reports it on /rudra/obstacle_guard.
StatusMeaning
clearForward path is clear.
slowingObstacle is in the slow-down zone.
blockedObstacle is inside stop distance.
reverse_or_turnCommand is reverse or turn-in-place, so it is allowed through.
no_fresh_scanNo recent scan is available.
disabledGuard is intentionally bypassed.

The guard does not start the LiDAR for you. If the driver is down, forward commands pass through based on the node's no-fresh-scan behavior. Treat the sensor and the motor filter as separate systems.

Safety and Shutdown

  • Lift the wheels for the first motor test after any firmware or bridge change.
  • Keep battery power easy to disconnect.
  • Close Arduino Serial Monitor before launching ROS.
  • Only one bridge may own the Teensy serial port at a time.
  • Start with low-speed commands when validating a new change.
  • Do not move on to autonomy until manual drive, LiDAR, and obstacle guard behavior are boringly reliable.

Practical shutdown order:

  1. Release the controller sticks or stop publishing /cmd_vel.
  2. Stop the ROS launch.
  3. Run bash scripts/stop_lidar.sh if the LiDAR driver is running.
  4. Remove Sabertooth battery power.
  5. Shut down the NUC cleanly.
sudo shutdown now
